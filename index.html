01
Outcome

It shipped. That is the part that matters.

Accessible Places answers one question for a wheelchair user: can I actually get into this place? It merges accessibility data from six providers, resolves the same venue appearing in several of them, and attaches a confidence score to every claim — because the honest answer is often “the sources disagree”.

Why an executive should care about this section

The hardest part of AI product work is not the demo. It is closing the loop with real users fast: 43 tracked issues were triaged and shipped in days, not quarters, several of them raised directly by production error telemetry rather than a support queue. It is also surviving upstream reality — free data providers that rate-limit aggressively, or return an HTTP 200 with a truncated HTML page instead of an honest error when overloaded — which is why the search pipeline races providers in parallel, retries with backoff, and degrades gracefully instead of failing outright. And it is the unglamorous question of what the product shows when the data is simply missing. All of that was met and solved here.

So what Closing that loop end to end — user report to root cause to shipped fix, under real rate limits and real upstream failures — builds a sharper instinct for risk assessment and complexity estimation than reading about it does, and the judgement to know when an AI-generated answer can be trusted and when it needs a human to check it.

03
Frontier

Deliberately working past the comfortable edge

Most AI-assisted development stops at autocomplete. The practices below go considerably further, and each one was adopted because a real problem demanded it — which is the only way anyone learns where the limits genuinely are.

Practices

Agents that delegate to agents

58 sub-agent delegations: a lead agent decomposing work and dispatching specialised agents for research, planning and review, then integrating their findings. Multi-agent orchestration in daily practice.

Agents that drive a real browser

571 browser-automation calls. Agents opened the running app, clicked through it, read the console and verified their own fixes end to end — closing the loop between writing code and confirming it works.

A written context system

A ~200-line architecture briefing maintained specifically so agents inherit hard-won project knowledge instead of rediscovering it. Context engineering as an explicit, maintained artefact.

Quality gates agents cannot bypass

Automated accessibility and colour-contrast checks that fail the build, plus a pre-commit hook running the full test suite. Guardrails designed on the assumption that generated code needs verifying.

Self-hosted infrastructure

A dedicated server running a private map-data API, geocoder and error tracker — provisioned after public services proved too unreliable for a product people would depend on.

Scheduled autonomous work

Recurring agent jobs that validate data coverage and refresh caches nightly, plus long-running background tasks. Agents operating unattended, on a schedule.

The failure that taught the most

A deep-link bug survived seven consecutive attempted fixes across versions 10.31 to 10.38. It only fell once error telemetry was wired into the production app and the actual event ordering became visible — the root cause was a race in component start-up that no amount of reasoning about the code had revealed. The lesson generalises directly: agents are extraordinary at generating plausible explanations, and plausible is not the same as correct. Observability is what converts an AI-built system into a maintainable one.

So what The limits of these tools are known here from having hit them, which is the only credible basis for setting expectations with a board or a customer.

On the honesty of the 300-hour figure

Hours were not recalled or estimated. For the period where complete event-level logs exist, working time was reconstructed directly from timestamps. For the earlier period only prompt-level records survive, so the same clustering method was applied there — and then validated against the overlapping weeks where both sources exist. On that overlap the prompt-based method returned 101.0 hours against 104.3 measured: a 3% deviation. One inflated-looking metric was discarded outright — a logged “longest session” of 261 hours, which counts idle time and means nothing.

So what Presented with its own error bars and its own discarded data points. That instinct is the same one that separates a trustworthy product metric from a vanity one.

Responsible AI, as an enforced gate rather than a principle

The product targets WCAG 2.2 AA and is itself an accessibility tool, so the standard had to be held. Automated checks for screen-reader structure and colour contrast run on every change and fail the build if violated. The honest limits are documented too: what automated testing genuinely cannot verify — contrast over map imagery, screen-reader behaviour on real devices — is written down as requiring human testing rather than quietly claimed as passing.

So what Ethics implemented as a blocking gate, with its own limitations stated openly. That is what responsible-AI governance looks like when it is real rather than declared.
06
Motivation

Nobody paid for this, and nobody was meant to

The product is free, carries no advertising, has no revenue model and collects no personal data. It exists because the alternative — a wheelchair user arriving at a café to find one unmarked step at the door — is a solvable problem that nobody had quite solved for the German-speaking region.

Impact
  • The problem is specific. Not “is this place nice” but “can I get through the door, is there a toilet I can use, is there a parking space I can use” — answered separately, because they fail separately.
  • The data was scattered. Six providers each held part of the answer, in incompatible formats, with no single place to ask. Merging them was the actual product insight.
  • Uncertainty is shown, not hidden. When the sources cannot confirm accessibility, the app says so and offers a way to contribute better information rather than guessing.
  • It goes where the users are. Native apps on both mobile platforms, because expecting people to find a website when they are already standing outside the building is a design failure.

Why this belongs in a hiring conversation

Three hundred hours of unpaid evenings and weekends went into a product whose only return is that some people can leave the house with more confidence. That says something about motivation that no interview answer can. It also happens to be the ideal training ground for AI product leadership: a domain with messy, contradictory real-world data, users who cannot afford a wrong answer, and a hard ethical requirement built into the definition of done.

So what The judgement on display here was formed under real constraints and real stakes, not in a sandbox.

Method
& sources

Where the numbers come from. Four independent sources, combined: the version-control history of the repository (commits, code volume, timing); complete event-level agent logs including per-message token accounting; a prompt-level history covering the full period; and daily aggregate usage records. No figure on this page is self-reported.

How working time was derived. Activity was grouped into sessions by separating blocks with genuine gaps in activity. Where complete event logs exist the result is a direct measurement; for the earlier period the same method was applied to prompt records and calibrated against the overlapping five weeks, where it deviated by 3%.

Attribution and caveats. The project was renamed mid-flight, so records under both names are merged. Machine-wide token totals are apportioned to this project by its share of logged instructions — 95% over the period. Token counts are input plus output, matching how the tooling itself reports usage; cached context re-read is tracked separately and reported apart from the headline figure, because it measures context carried rather than new work produced.
